The
UnicodeDecodeError with pipreqs occurs when it encounters a file with an encoding other than the expected UTF-8. The best solutions involve either specifying the correct encoding with a command-line flag or ignoring directories that contain problematic, non-UTF-8 files (like virtual environment folders). 
Solutions

    Specify the encoding (Recommended): If your project uses a different consistent encoding, you can tell pipreqs to use it with the --encoding flag. A common alternative on Windows is 'ISO-8859-1' or 'windows-1252'.
    bash

    pipreqs --encoding=ISO-8859-1 /path/to/project

    You may need to experiment to find the correct encoding.
    Ignore problematic directories: The error often occurs in virtual environment directories (.venv, env, etc.) which contain non-Python files or files with different encodings. You can use the --ignore flag to skip these.
    bash

    pipreqs /path/to/project --ignore .venv,env

    This approach is highly effective and recommended by users on GitHub.
    Identify and fix the file: If you can locate the specific file causing the error (the error message usually provides the position, which can help narrow it down), you can open it in a text editor (like VS Code, Notepad++) and change its encoding to UTF-8 and save it.
        On Linux/macOS, you can use file -i <filename> or file -I <filename> to try and identify the file's current encoding.
    Move code to a new folder: As a last resort, you can create a new, clean folder, move only your source code files into it (leaving out hidden environment folders), and run pipreqs from that new location.

new info: Streamlit recursively monitors changes in the parent folder of the application script, e.g. app.py [source].

The idea is that if source files update, the app is 'rerun', as a convenience for app development. However, if this folder contains too many subfolders and subfiles, apparently watchdog reaches a limit and raises an error. This easily happens if you run straight from your home directory, which contains tons of configuration files, etc. Or maybe when you run from a git repository with numerous files in .git (just speculating).

You can resolve this by either placing the app.py script in a (sub)folder than only contains dependencies that you'll edit during development, or by --server.fileWatcherType none and accept the nuisance of manually restarting.

streamlit run app.py --server.fileWatcherType none


The
UnicodeDecodeError with pipreqs occurs when it encounters a file encoded in something other than UTF-8, which is its default assumption. The most direct fix is to specify the correct encoding when running the command or to ignore the problematic file/directory. 
Solutions
Here are the recommended solutions to resolve the error:

    Specify the encoding in the command: The simplest solution is to use the --encoding flag and set it to utf-8 (or the correct encoding of your files).
    bash

    pipreqs --encoding=utf8 ./your/project/path

    This often resolves the issue if your files are, in fact, UTF-8 but perhaps saved with a Byte Order Mark (BOM).
    Ignore the problematic file or directory: The error often happens when pipreqs scans files that are not part of your primary source code, such as virtual environment files (.venv, env) or data files (e.g., CSVs, log files) that use a different encoding.
    bash

    pipreqs --ignore ./path/to/ignore

    You can ignore your virtual environment folder to prevent this issue. For example:
    bash

    pipreqs --ignore .venv

    Combine flags: You may need to use both flags together for the best results:
    bash

    pipreqs --encoding=utf8 --ignore .venv

    Manually re-encode the file: If you know which file contains the invalid character, you can open it in a text editor like Sublime Text or Notepad++, change the encoding to UTF-8, and save it.
    Identify the actual encoding: If the above doesn't work, you can use a library like chardet in a Python script to try and determine the file's actual encoding. Once identified, you can either re-encode the file or use that encoding in the pipreqs command.
